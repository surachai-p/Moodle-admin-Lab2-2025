# ใบงานการทดลองที่ 2: การบริหารจัดการ Moodle LMS สำหรับผู้ดูแลระบบ

**วัตถุประสงค์:** เพื่อให้นักศึกษาสามารถ
1. อธิบายแนวทางการวางแผนโครงสร้างระบบ Moodle สำหรับสถานศึกษาได้
2. สร้างและจัดการ Category, Cohort อย่างเป็นระบบ
3. นำเข้าข้อมูลผู้ใช้จำนวนมากด้วย Bulk upload ได้
4. กำหนดค่าเริ่มต้นที่สำคัญสำหรับการใช้งานจริง
5. บริหารจัดการสิทธิ์และความปลอดภัยได้

-

##  แนวคิดในการเลือกวิธีเพิ่มข้อมูลแบบ Manual หรือ CSV Upload

### การสร้าง Category
- **แนะนำ:** สร้างแบบ **Manual** ทุกกรณี
- **เหตุผล:**
  - จำนวนไม่มาก (10-30 รายการ)
  - ต้องการความระมัดระวังในการออกแบบ
  - Moodle ไม่มี built-in CSV upload

### การสร้าง Cohort
- **< 20 รายการ:** เลือกแบบ Manual (ควบคุมง่าย)
- **> 20 รายการ:** เลือกแบบ **CSV Upload** 
  - ประหยัดเวลามาก
  - ลดความผิดพลาด
  - สำหรับโรงเรียนในใบงาน (24 ห้อง + 8 กลุ่มสาระ) ควรใช้ CSV

### การสร้าง Users
- **แนะนำ:** **CSV Upload** ทุกกรณี
- มีประสิทธิภาพและจัดการง่าย

## 📚 ส่วนที่ 1: ทฤษฎีและแนวคิดพื้นฐาน

### 1.1 Moodle Architecture สำหรับสถานศึกษา

![Moodel System Architectuer](images/moodle-system-hierrachy.png)

**ความสำคัญของการวางแผนโครงสร้างของระบบ:**
1. Scalability - ขยายระบบได้ง่าย
2. Maintainability - บำรุงรักษาสะดวก
3. User Experience - ใช้งานง่าย
4. Security - มีความปลอดภัย
5. Reporting - วิเคราะห์ข้อมูลได้

### 1.2 Categories (หมวดหมู่รายวิชา)

**ตัวอย่างโครงสร้าง Categories:**

```
ปีการศึกษา 2568
└── ภาคเรียนที่ 1
    ├── มัธยมศึกษาตอนต้น
    │   ├── ชั้น ม.1
    │   ├── ชั้น ม.2
    │   └── ชั้น ม.3
    └── มัธยมศึกษาตอนปลาย
        ├── ชั้น ม.4
        ├── ชั้น ม.5
        └── ชั้น ม.6
```

**Best Practices:**
- ใช้การตั้งชื่อที่สอดคล้องกัน
- จำกัดระดับความลึกไม่เกิน 3-4 ระดับ
- ใช้รหัส ID number เช่น AY2568-S1-M1

### 1.3 Cohorts (กลุ่มผู้ใช้)

**ประเภทของ Cohorts:**
1. System Cohorts - ใช้ได้ทั้งระบบ
2. Category Cohorts - จำกัดในแต่ละ category

**ตัวอย่างการตั้ง Cohorts:**
- นักเรียน-ม.1-1, นักเรียน-ม.1-2, ...
- ครู-คณิตศาสตร์, ครู-วิทยาศาสตร์, ...
- ผู้บริหาร, หัวหน้ากลุ่มสาระ

**ประโยชน์การใช้ Cohorts:**
- Bulk Enrollment (ลงทะเบียนเป็นกลุ่ม)
- Automated Enrollment (อัตโนมัติ)
- Easy Management (จัดการง่าย)

## ขั้นตอนการทดลองปฏิบัติการ
### 1: การตั้งค่าเริ่มต้นของระบบ

#### 1.1 Login เป็น Admin

```
URL: http://localhost
Username: admin
Password: [รหัสที่ตั้งไว้]
```

#### 1.2 ตั้งค่าพื้นฐานต่าง ๆ ดังนี้

**1) ภาษาและเขตเวลา:**
- Site administration > Language > Language settings
- Default language: ภาษาไทย (th)
- Default timezone: Asia/Bangkok

**2)Password Policy:**
- Site administration > Security > Site security settings
- Password length: 8
- Digits: 1, Lowercase: 1, Uppercase: 1
- Force password change on first login: Yes

**3) ตั้งค่าการลงทะเบียน:**
- Site administration > Plugins > Authentication > Manage authentication
- ปิด Email-based self-registration (เลือกเป็น Disable)

**4)ตั้งค่่า Automated Backup:**
- Site administration > Courses > Backups > Automated backup setup
- Active: Enabled
- Schedule: 02:00 AM
- Keep: 7 days
- Include user data: Yes

## 3.2. Upload Cohorts

### ตัวเลือกการ Upload Cohorts

| Upload Mode | ความหมาย | ใช้เมื่อ |
|-------------|----------|---------|
| `Create new cohorts only` | สร้างใหม่เท่านั้น ข้ามถ้ามีอยู่แล้ว | ✅ ครั้งแรก |
| `Create or update cohorts` | สร้างใหม่ + อัพเดตที่มีอยู่ | อัพเดตชื่อ / description |
| `Update existing cohorts only` | อัพเดตเท่านั้น ไม่สร้างใหม่ | แก้ไขข้อมูล cohort |

### ตัวเลือกอื่น

```
CSV settings:
├── CSV delimiter: Comma / Semicolon / Colon / Tab
└── Encoding: UTF-8 / Other
```

> ⚠️ **สำคัญ:** การ Upload Cohorts เป็นการสร้าง "กลุ่มเปล่า" ไม่ได้ใส่สมาชิก

> - ระบุ `cohort1=STD-M1-1` ใน CSV ตอน Upload Users
> - เพิ่ม manual ที่ Cohorts > Assign

- เลือกไฟล์ CSV
- เลือก Upload Mode
- เลือก CSV delimiter: Comma
- เลือก Encoding: UTF-8

#### 4.2 Upload Users

### Upload Type (ตัวเลือกสำคัญที่สุด)

| Upload Type | ความหมาย | ใช้เมื่อ |
|-------------|----------|---------|
| `Add new only, skip existing` | เพิ่มเฉพาะคนใหม่ ข้ามถ้ามีอยู่แล้ว | ✅ เริ่มต้นระบบครั้งแรก |
| `Add all, append number if needed` | เพิ่มทั้งหมด ถ้า username ซ้ำให้เพิ่มตัวเลขที่ด้านหลัง เช่น john2 | ⚠️ ระวัง username ซ้ำ |
| `Add new and update existing` | เพิ่มคนใหม่ + อัพเดตข้อมูลคนเดิม | ✅ อัพเดตข้อมูลกลางปี |
| `Update existing users only` | อัพเดตคนที่มีอยู่เท่านั้น ไม่สร้างใหม่ | Reset password / เปลี่ยน cohort |
| `Delete incoming users` | **ลบ** ผู้ใช้ที่อยู่ในไฟล์ | ⛔ อันตราย ระมัดระวังในการใช้ |
| `Allow deletes` | อนุญาตให้ลบได้ (ใช้ร่วมกับ option อื่น) | ⛔ ระมัดระวังในการใช้ |


## คำถามท้ายการทดลอง

**1.** อธิบายความแตกต่างระหว่าง Category และ Cohort
คือการจัดกลุ่มคอร์สเพื่อโครงสร้างเนื้อหา เช่น แบ่งตามคณะหรือหัวข้อ ส่วน Cohort คือกลุ่มผู้ใช้ที่ admin สร้างขึ้นเพื่อจัดการสมาชิกในระดับ site หรือ category โดย Category ตอบคำถาม "เนื้อหาอยู่ที่ไหน" ส่วน Cohort ตอบคำถาม "ใครเข้าถึงอะไรได้บ้าง"

**2.** เหตุใดควรใช้ Cohort sync
เมื่อ enroll Cohort เข้า course แบบ sync ระบบจะอัปเดต enrollment อัตโนมัติทุกครั้งที่มีการเพิ่มหรือลบสมาชิกออกจาก Cohort ทำให้ admin ไม่ต้องจัดการ enrollment ทีละ course ประหยัดเวลามากเมื่อมีผู้ใช้จำนวนมาก และลดโอกาสเกิด human error จากการ enroll ด้วยมือ

**3.** Best practices สำหรับ Category structure คือแบบใด
ควรออกแบบลำดับชั้นไม่เกิน 3 ระดับ (เช่น คณะ → ภาควิชา → รายวิชา) เพื่อไม่ให้ซับซ้อนเกินไป ตั้งชื่อให้สื่อความหมายและสม่ำเสมอ กำหนด role และ permission ในระดับ category เพื่อให้ subcategory และ course รับ permission ต่อโดยอัตโนมัติ และหลีกเลี่ยงการสร้าง category ที่มี course เดียว เพราะสิ้นเปลืองโครงสร้างโดยไม่จำเป็น
